The three trust boundaries that matter 1. Server → Model (metadata channel). Tool names & descriptions are injected into the model's context as authoritative text. Tool poisoning, preference manipulation, and shadowing live here.
2. External data → Model (output channel). Whatever a tool returns (a GitHub issue, a web page, a DB row) is fed back to the model. Indirect prompt injection lives here.
3. Model/Server → System (execution channel). The server runs with the host's privileges and no protocol-level isolation. Command injection, sandbox escape, credential theft live here.

The MCP server lifecycle (4 phases, 16 activities)

Hou et al. model an MCP server's life as four sequential phases. Crucially, a threat's origin phase and its trigger phase often differ — e.g. tool poisoning is authored at creation but fires at operation. Reasoning from origin is what tells you whether a static scan can catch it.

PhasePrimary actorKey activitiesSecurity relevance
CreationDeveloperMetadata definition · capability declaration · code implementation · slash-command definitionMost vulnerabilities are introduced here. Capability declarations & descriptions are the poisoning surface → highest-yield target for a static scanner.
DeploymentDeveloper → User / Clientserver release · installer · environment setup · tool registrationSupply-chain & installer-spoofing risks; provenance and signature checks belong here.
OperationUserintent analysis · external-resource access · tool invocation · session managementRuntime-only threats: indirect prompt injection, tool-chaining abuse, session/auth attacks. This is the proxy's domain.
MaintenanceDeveloperversion control · configuration change · access audit / loggingRug pulls, vulnerable versions, configuration drift, privilege persistence.

Attacker archetypes

Type 1

Malicious Developer

Authors a server/tool that is hostile by design — poisoned metadata, typosquatted name, hidden backdoor, or a planned rug pull. Origin: creation.

Type 2

External Attacker

No control over the server, but poisons what it touches — a tampered installer, or adversarial content in a data source the tool fetches (indirect prompt injection).

Type 3

Malicious User

A legitimate user (or a compromised agent acting as one) abusing tools at runtime — credential theft, sandbox escape, tool-chaining, session hijacking.

Type 4

Security Flaws

No active adversary required — latent risk from vulnerable dependency versions, retained privileges, or drifted misconfiguration.

Takeaway for the scanner design Of the 16 threats, roughly half carry a strong static signal (1–4, 7, 8, 14 outright; 6, 10, 11, 15, 16 partially). That is a defensible, well-scoped target for the Hauptprojekt. The remaining runtime-only threats (5, 9, 12, 13) become the justification for the Masterarbeit proxy — your two artifacts cover complementary halves of the same taxonomy, which is a clean thesis narrative.
