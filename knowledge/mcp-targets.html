MCP Target Sets for Scanner Evaluation

Working list of MCP servers and benchmark groups we can use for the contract-derived sandbox scanner. This page focuses on practical setup difficulty, not only paper relevance.

Current recommendation Use Connor as the first malicious benchmark because it ships runnable MCP server directories and configs. Use the easy subset of Connor benign and later MCP-Bench as benign/utility controls. Treat MSB as a later agent-attack benchmark, not as the first scanner target set.

Evaluation ladder

StageSample sizeTarget mixWhy this size
Smoke 8-10 MCPs 4-5 Connor malicious + 4-5 easy benign servers Validates Docker startup, MCP handshake, contract generation, trace capture, and verdict output.
First presentation 20-25 MCPs 10-15 Connor malicious + 10 easy benign Enough for early detection-rate and false-positive discussion without spending the whole project on dataset plumbing.
Good Hauptprojekt result 40-60 MCPs 20-30 malicious + 20-30 benign Credible scope for a 4-week implementation/writeup cycle if the harness is stable.
Stretch 100+ MCPs Connor PoCs + Connor benign + MCP-Bench no-key servers Useful later, but only after setup failure handling, timeouts, and report generation are reliable.

Connor target set

Connor is the strongest starting point because the released repo contains MCP server directories with mcp.json startup configs. The detector itself also uses Docker, and the repo explicitly warns that the PoCs may have side effects.

Connor PoCs

The repo has 114 PoCs: 6 attack-goal directories times 19 influence paths. Each PoC directory contains mcp.json and pyproject.toml.

Why this set It balances malicious metadata attacks, hidden behavior, weather/API-looking attacks, local computation, file access, system-info access, stateful todo behavior, and benign network/search behavior. It is also mostly Python/uv, which keeps the first harness implementation narrow.

Notes for implementation

  • Normalize every target into one internal schema: id, source, path, command, args, transport, label, expected_setup.
  • Connor configs use /app/... paths. The Docker runner should mount/copy each target directory to /app/<target> or rewrite paths before launch.
  • Python 3.13 support matters for Connor. Use a sandbox image with uv and Python 3.13 available.
  • Treat setup failure as a result category, not as a crash: setup_failed, handshake_failed, timeout, needs_credentials.
  • For samples that legitimately require network, run a second mode with controlled egress logging. Default runtime should be network disabled.
