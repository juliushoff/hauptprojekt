Positioning Update — Connor & the Pivot to Runtime Enforcement

Session of 2026-06-10. A literature re-check (the KB sweep was last run 2026-05-30) surfaced papers that partially scoop the original "behavioral divergence detection" framing. The headline: detection of declared-vs-actual divergence is now taken (statically and dynamically); runtime enforcement of a derived contract — plus drift/rug-pull and cross-server — is still wide open, and is explicitly named as future work by the very paper that scooped us. This page supersedes the framing in Gap Analysis; the architecture in Design & Scope and Build Plan is unchanged — only the headline claim moves.

One-line takeaway Stop pitching a divergence scanner. Pitch a contract-based runtime-enforcement system that consumes divergence detection (ours or others') and does the runtime half nobody has built: deterministic per-call enforcement + drift/rug-pull + cross-server. Same proxy, same contract, same datasets — repositioned as the headline instead of the sequel.

1. The paper that forced this: Connor

★ From Component Manipulation to System Compromise: Understanding and Detecting Malicious MCP Servers — Huang et al., Fudan University; arXiv v2, May 2026 (orig. Apr 2026); submitted to ACM TOSEM. Builds the first component-centric PoC dataset of 114 malicious MCP servers and a detector named Connor. arXiv:2604.01905

What Connor is: a two-stage offline, dynamic detector whose core principle is — in their own words — "behavioral deviation from function intent." That is our divergence idea.

  • Pre-execution: a Config Analyzer scans the startup shell command for malice; an Intent Inspector reads each tool's description + schema and extracts its "function intent," explicitly separating legitimate intent from injected/poisoned instructions. That sanitized intent is the behavioral baseline.
  • In-execution: it actually runs the server in a simulated host, auto-generates intent-aligned user queries, traces execution (code slicing + an LLM "Code Semantic Generator"), and an LLM "Behavior Deviation Judger" flags steps where observed behavior deviates from the declared intent.
  • Output: a per-server verdict (malicious / benign) + rationale. F1 = 94.6% (beats mcp-scan / AI-Infra-Guard by 8.9–59.6 pts). LLM cost ≈ $48.6 per evaluation run. Real-world scan of 1,672 marketplace servers found 2 malicious.
  • Dataset + code are public — this is the gift (see §6).
  • It even names the proxy deployment: "Connor can be deployed as a runtime proxy on the real host" — but for detection/observability, not enforcement.
What Connor does NOT do — this is the whole thesis
  • No enforceable contract artifact. It emits a verdict, not a machine-readable, pinned capability envelope a runtime layer consumes.
  • The "judge" is an LLM, not a deterministic gate. Slow, $48/run, carries the Firewall-Fallacy fragility. No ms-latency per-call enforcement.
  • No drift / rug-pull over time. One-shot audit. Once it says "benign," it walks away — a server that mutates after the audit is structurally invisible to it (their own Limitation: "attacks whose malicious logic is not exercised at runtime may evade detection").
  • No cross-server / inter-server analysis. Verbatim: "we do not consider inter-server attacks … left to future work."
  • No deterministic capability envelope + sandbox profile. It flags; it does not block an out-of-envelope call or confine the server.
And the kicker (their Limitations §): "a promising direction is to combine our proxy-based analysis with complementary static detectors to strengthen defenses." Connor names our direction as future work.

Takes the terms "behavioral contracts + runtime enforcement" — but contracts are hand-authored formal specs (P,I,G,R), generic agents, not MCP, not behavior-derived. A small, high-quality reusable testbed; baseline/hardened pairing is ideal for false-positive measurement. arXiv:2604.21477

3. The repositioning — detection is taken, enforcement is open

Both halves of detection are now claimed: static description-code inconsistency (2602.03580) and dynamic behavioral deviation (Connor). What none of them build is the deployable other half.

The new thesis claim (survives everything published as of 2026-06-10) Divergence detection exists — statically (2602.03580) and dynamically (Connor). What does not exist is turning that signal into a machine-readable, pinned capability contract and enforcing it deterministically at runtime, with drift/rug-pull detection and cross-server coverage. We build the enforcement layer these detectors call for. Detection becomes a means (reuse Connor's intent extraction or cite it as the detection baseline); the contract + deterministic enforcement + drift becomes the novelty.

One-line differentiators (related-work section)

  • vs Connor (2604.01905): Connor detects behavioral deviation with an LLM judge and emits a verdict, once, offline. We derive a pinned, machine-readable contract and enforce it deterministically at runtime, adding drift detection and sandbox confinement Connor does not provide — and catching the server that goes bad after the audit, which Connor by construction cannot.
  • vs "Don't believe everything you read" (2602.03580): they measure static description-code inconsistency across the ecosystem; we make that inconsistency enforceable as a runtime contract.
  • vs ABC (2602.22302): their contracts are hand-authored formal specs for generic agents; ours are behavior-derived, MCP-specific, and enforced on the live tool-call channel.

5. The two-axis framing & the scope dial (drift + cross-server)

If the professor asks for more scope, the Masterarbeit scales cleanly because drift and cross-server are the same machine on two axes:

  • Temporal axis — drift / rug-pull: "does this server still behave like it did at audit time?" Lower-risk; a complete thesis on its own; the must-have core.
  • Spatial axis — cross-server: "does this server behave correctly given the other servers loaded alongside it?" Harder (must model how A's output enters context and steers a call to B); additive depth.
Sequencing rule — avoid two half-built halves Build order is additive, never load-bearing on the hard part: (HP) scanner + contract derivation (single server) → (MA core) proxy + enforcement + drift → (MA depth) cross-server. If time runs out, cross-server is a strong evaluated extension or a clearly-scoped future-work chapter — not a hole. Pitch drift-focused enforcement as Plan A; keep "1+2" ready as the answer to "is that enough for a master's?" — phrase to Bettina: "the scope can scale from single-server drift to the multi-server / cross-server case, whichever depth fits."

6. Benchmarks & evaluation design

The trap — never lead with detection F1 A static-derived contract used as a detector will likely score below Connor's 94.6% F1. Running their dataset and reporting detection F1 enters a race we're not running, and invites "why is yours lower?" Detection F1 is never the headline. Mention it once, honestly, then pivot to enforcement metrics.

Live-enforcement evaluation is established practice — AEGIS, MiniScope, MCP Guardian, Prompt-CFI all report a (block-rate / FP / latency) triple, so we are not in a vacuum.

The self-built benchmark — legitimate, not circular (3 rules)

Drift and cross-server have no existing benchmark (nobody enforces over time / across servers), so we construct one — and benchmark construction is itself a first-class contribution here (MCPTox, MSB, Pitfall Lab, Connor's set are all "we built a dataset" papers). The circularity objection ("you built a test you pass") is killed by three rules:

  1. Threat-derived, mechanical generation — not hand-picked. Define general mutation operators (inject undeclared egress; swap a tool's description post-audit; add a write to a read-only tool; trigger payload on call N) and apply them mechanically to the public benign+malicious corpora. The cases reflect the threat, not our solution.
  2. Run other systems on it. On every benchmark run 2–3 baselines under identical conditions: no-defense (floor), a generic proxy (regex/WAF ≈ Guardian), and Connor / mcp-scan where applicable. When they score ~0 on drift because they structurally can't and we score well, it's a capability gap exposed across systems, not a self-serving demo. This single discipline also guarantees we never report a number without a comparison generated in our own setup.
  3. Release it publicly — "first drift/rug-pull benchmark for MCP" becomes a citable side-contribution and signals good faith.
The real validity risk to neutralize in the methodology Same person building both attacks and defense can bias attacks toward catchable ones. Mitigate: (a) mechanical mutation operators, not hand-picked cases; (b) reuse independent attack sources we didn't write — Connor's PoCs, MCP-ITP's automated poisoning (2601.07395), Pitfall Lab variants — for as much of the corpus as possible. State this explicitly and the objection is gone.

Corpus reuse — we build very little ourselves

  • Connor's 114-server PoC dataset + code (public) — large, labeled, peer-reviewed malicious-server corpus (data leakage, reverse shell, ransomware, backdoor, sabotage; single + multi-component).
  • Pitfall Lab — 6 baseline/hardened variants + P1–P6 + trace validators.
  • MCPTox / MSB — protocol-level attack instances for the live proxy eval.
  • MCP-Universe / MCP-Bench — benign tasks for utility/FP.
  • Hand-craft only a handful of servers to isolate the specific contract/drift cases the public sets don't cover.
  • Caveat: MCPTox/MSB mostly poison the description against benign-code servers (metadata-poisoning). For the scanner's code-divergence detection you need code-divergent servers → Connor's dataset + Pitfall Lab provide those.
