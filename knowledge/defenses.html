Defenses

Mitigations organized as four defense layers, then mapped per-attack. The layering matters: it shows where your CLI scanner and proxy sit relative to everything else, and what they should not try to solve.

Four defense layers

L1 · Design-time

Static analysis & vetting

Scan manifests, descriptions, code, deps before a server is trusted. ← Hauptprojekt lives here. Pattern + taint + NL-instruction detection, secret & CVE scanning, identity checks.

L2 · Supply-chain

Identity & integrity

Signed manifests, verified publishers, namespace governance, version pinning, reproducible builds, checksum/signature on install. Makes typosquatting, installer spoofing, rug pulls detectable.

L3 · Runtime

Interception & policy

A proxy / guardian between client and server inspects calls & responses, enforces least privilege, screens tool output, sandboxes execution. ← Masterarbeit lives here.

L4 · Model-level

Robust agents

Train/align the model to resist injection (SecAlign, StruQ), separate data from instructions, prioritize the system channel (Prompt-CFI). Out of your build scope, but cite as the complementary track.

The "Firewall Fallacy" — read this before building guardrails MCP-TDP (2605.24069) found that naïve prompt-based guardrails are not just ineffective against tool-description poisoning — they can be counterproductive, because adding more natural-language "be careful" instructions gives the attacker more surface to manipulate and can degrade utility. Implication: prefer deterministic / structural defenses (signatures, allow-lists, taint analysis, schema enforcement, sandboxing) over "ask the LLM nicely". A guardrail LLM is a layer, not the layer.

Notable concrete defense proposals (to cite / compare)

  • ETDI (2506.01333) — OAuth-enhanced, cryptographically-signed tool definitions + policy-based access control; directly targets tool squatting & rug pulls. A good L2 baseline to discuss.
  • MCP Guardian (2504.12757) — a security-first layer (auth, rate-limiting, logging, scanning) wrapping MCP traffic; closest "guardian" framing to your proxy.
  • MiniScope (2512.11147) / over-privilege auditing (2603.21641) — least-privilege authorization for tool-calling agents.
  • Reactive Self-Correction (MCP-TDP) — agent detects & reverts its own malicious action post-execution; interesting but reactive (damage may already be done) — contrast with proactive interception.
  • Prompt Control-Flow Integrity (2603.18433) — priority-aware runtime defense enforcing that system/user instructions outrank injected ones.
  • SecAlign (2410.05451), StruQ (2402.06363) — model-level injection robustness; SecAlign reports <10 % injection success.
Design principle for both your artifacts Favor deterministic, structural, verifiable controls (signatures, schemas, allow-lists, taint graphs, sandboxes) as the backbone; use an LLM judge only as an additional recall layer with calibrated thresholds — never as the sole gate. Always report the security/utility trade-off (NRP-style), because over-blocking is the failure mode reviewers will probe.
