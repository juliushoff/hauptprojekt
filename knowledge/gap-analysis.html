Design, now evidence-backed: deterministic spine + small LLM on top Deterministic contract checks are fast and near-perfect on what they cover but blind to novelty (AEGIS 100%/8.3 ms but misses unseen; PCFI 0% APR/0.04 ms but brittle to paraphrase). A small model covers the semantic/intent gap but is fragile alone (SentinelAgent's NLI: great → 13% TPR under paraphrase). Therefore: the contract is enforced deterministically; the small LLM/NLI is a secondary signal only for divergence severity and intent-consistency — never the gate. This is the "stochastic-deterministic boundary" of 2605.20173 applied to security.

Verdict on the three candidate directions

OPEN ✓

B — Behavioral divergence

"Does the tool actually do what its description/schema claims?" Every neighbor leaves this open; mcp-sec-audit explicitly declines it; MiniScope assumes it; MCP-ITP proves static can't do it. Strongest standalone gap.

OPEN ✓

B+A — Contract + enforcement

Scan→contract→enforce as one loop is unbuilt. MiniScope has enforcement (doc-derived), Guardian has a generic proxy, mcp-sec-audit emits a report. Nobody emits a behavior-derived contract and enforces it. Best HP/MA split.

CROWDED ⚠

E — Ecosystem measurement

VIPER-MCP (39,884) + First Glance (1,899) own the static-prevalence space. Survives only as a novel-lens slice (behavioral/dynamic prevalence) — i.e. a scale-up evaluation of B, not the headline.

Recommended thesis: Behavioral Capability Contracts for MCP Tools

One coherent idea that absorbs B, B+A, and the surviving slice of E, differentiates cleanly from every neighbor, and answers the survey's own §6.2 gaps (missing monitoring; "config drift / payload corruption undetected until after exploitation"; no runtime sandboxing).

Hauptprojekt — the behavioral scanner (CLI) Execute each MCP tool in an instrumented sandbox with probe inputs; observe actual behavior (network / filesystem / process-exec / data-flow). Turn the tool's declared semantics (description + JSON schema) into an expected capability set, and report divergence: a "calculator" that opens a socket, a "read" tool that writes, undeclared egress. Emit a machine-readable capability contract (the allowed behavioral envelope + a pinned manifest/code hash). Original character without needing large results — a divergence demo on a handful of poisoned vs. benign servers is enough.
Masterarbeit — the enforcing proxy + drift + intent A runtime proxy that (1) enforces the contract on live calls (block out-of-envelope behavior), (2) detects drift / rug-pulls (behavior diverging from the pinned contract over time), and (3) adds intent-consistency checks that catch MCP-ITP-style implicit poisoning (a legit high-privilege tool invoked against the user's request). Empirical eval: detection on MCPTox + MSB + the implicit-poisoning subset where static scanners hit 0.3%; utility/regression on MCP-Universe; report an NRP-style security/utility trade-off; plus a behavioral-prevalence scan of a public-server sample (the open slice of E).

One-line differentiators (for your related-work section)

  • vs mcp-sec-audit: divergence is defined against the tool's own declared semantics, and the output is an enforceable contract, not a risk report.
  • vs MiniScope: the contract is behavior-derived and fine-grained, and you verify the "tools honor their specs" assumption MiniScope is forced to trust.
  • vs MCP Guardian: enforcement is contract-driven and catches a compromised/poisoned tool — the case Guardian states it cannot handle.
  • vs VIPER-MCP: you target semantic divergence (incl. logic/over-privilege) and a deployed monitor, not code-level taint with a one-shot exploit oracle.
Honest risks to scope around (1) Sandboxing real tools is the hard part — tools needing live credentials or external services break naive harnesses (VIPER and mcp-sec-audit both hit this). Plan for mocking / recorded-dependency / containerized fixtures; solving it cleanly is itself a contribution. (2) "Declared semantics" from NL is fuzzy — turning a description into an expected-capability set likely needs an LLM, which reintroduces the Firewall Fallacy risk; mitigate by anchoring on the structured schema + observed behavior and treating the LLM as one signal, not the gate. (3) Coverage — probe inputs may not trigger conditional malicious branches (rug-pulls keyed on time/flags); be explicit that the proxy's continuous monitoring is what closes this, and don't over-claim static coverage.
