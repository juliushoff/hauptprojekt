3. Tech stack (concrete)

  • Language: Python — the MCP SDK, static-analysis tooling, and ML ecosystem all live here.
  • MCP: official mcp Python SDK for the harness and the proxy transport.
  • Static analysis: custom AST walker (MVP) → Bandit / Semgrep / CodeQL (depth).
  • Sandbox: Docker + seccomp; gVisor if you want stronger isolation; container network policy for egress.
  • Models: scan-time declared-extraction → a strong API model (offline, latency-free). Runtime intent-check → a small local model (Ollama/vLLM) for latency, cost, and privacy.
  • Contract: JSON + JSON Schema.

4. Three technical gotchas (decide these early)

#1 — Intent-consistency needs the user's task, which the proxy may not see The proxy sits at client⇄server and sees tool calls, not necessarily the original user prompt. To check "does this call match what the user asked," you need the task. Options: (a) the host/agent framework forwards the current task to the proxy (many expose it); (b) integrate the check into the host instead of a standalone network proxy; (c) in your eval harness you control everything, so you inject the intent. MVP: (c); productionizing needs (a). Flag this as a stated assumption.
#2 — Source-available vs running-only servers Static analysis needs the source. Open-source Python servers give you that. Remote/binary-only servers don't — there you fall back to declared-only + dynamic probing + runtime monitoring. MVP: assume open-source Python servers. Note remote/binary as future work.
#3 — Probe-input coverage (for dynamic analysis) Generating inputs that actually trigger a tool's code paths is non-trivial; use schema-based fuzzing + LLM-generated plausible inputs. Conditional malicious branches (rug-pulls keyed on time/flags) may not fire during probing — acknowledge the coverage limit, which is precisely why the runtime proxy complements the scanner.

The de-risk line M0–M4 is the entire Hauptprojekt and needs no sandbox, no proxy, no runtime. It is self-contained, demonstrable, and novel on its own. Everything from M5 is additive depth for the thesis — so an early failure in the hard parts can never sink the Hauptprojekt.

6. Out of scope (for the MVP — state it explicitly)

  • Non-Python servers; remote/binary-only servers (declared-only fallback).
  • Multi-server compositional analysis (cross-server shadowing across the full loaded set).
  • A full adversarial-robustness study of the detectors.
  • A fine-tuned specialized small model (stretch goal — see Gap Analysis).
