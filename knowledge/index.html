What is MCP, in one paragraph The Model Context Protocol (Anthropic, Nov 2024; now under the Linux Foundation's Agentic AI Foundation) is an open standard that lets an LLM agent discover, describe, and call external tools through a uniform interface. Architecture = Host (the AI app, e.g. Claude Desktop / Cursor) → Client (1:1 connector inside the host) → Server (exposes tools, resources, prompts). Tools are first-class, composable objects carrying natural-language metadata. That metadata is exactly what makes MCP powerful — and exactly what widens the attack surface.

The core security thesis

MCP turns tool descriptions, tool outputs, and tool composition into untrusted input that flows straight into the model's instruction channel. Classic software security assumes code is the threat; in MCP, natural-language metadata and data are the threat, and the "interpreter" is a probabilistic LLM that cannot reliably separate data from instructions. Two compounding facts from the benchmarks:

  • More capable models are more vulnerable. Tool poisoning exploits superior instruction-following — o1-mini reaches 72.8 % attack success on MCPTox; GPT-4o ≈100 % on six high-risk MCP-TDP scenarios.
  • Safety alignment barely helps. Agents almost never refuse these attacks (Claude-3.7-Sonnet refusal rate <3 % on MCPTox) because the malicious action is carried out through legitimate tools.

How this maps to your project

The single most useful organizing axis for your work is when an attack becomes observable. It cleanly splits Hauptprojekt scope from Masterarbeit scope.

PhaseWhat it can seeCatchesMisses
Hauptprojekt
static / pre-flight CLI scan		 Server manifest, tool/resource metadata, descriptions, declared schema, source & dependencies, config files, installer provenance. Tool poisoning text, preference-manipulation phrasing, typosquatting/name collisions, command-injection sinks, over-broad capability declarations, plaintext secrets, vulnerable deps. Anything that only appears at runtime: rug pulls (post-trust mutation), indirect prompt injection in live tool output, tool-chaining abuse, session/auth attacks.
Masterarbeit
runtime / live proxy		 Actual tool-call requests & responses, call sequences, data flows, cross-server interactions, dynamic metadata changes. Indirect prompt injection, rug pulls / dynamic mutation, cross-server shadowing in action, tool-chaining exfiltration, out-of-scope parameters, credential egress. Nothing structurally — but cost is latency, false positives, and needing a policy/decision model.

See Attack Vectors for the per-attack static / runtime / both classification — that table is the spec for what the CLI scanner should detect vs. what is deferred to the proxy.
