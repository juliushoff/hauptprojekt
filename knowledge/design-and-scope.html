Design, Feasibility & Scope

How a behavioral-capability-contract system actually works, whether it's buildable by one student, the latency budget, and a concrete Hauptprojekt / Masterarbeit split. The contract is the load-bearing artifact: the scanner produces it, the proxy consumes it.

1. The latency budget (why the split is natural)

Two components, two completely different timing regimes:

ComponentWhen it runsLatency budgetConsequence
Scanner (HP)Offline, pre-deployment, once per server/versionNone — seconds to minutes per tool is fine (VIPER: 926 s/server)Can afford heavy static analysis, sandbox probing, and LLM passes.
Proxy (MA)Online, on every tool callTight but relative — must be small next to the agent loop (seconds) & API round-trips (10–100 ms)Deterministic checks must be ~ms; the LLM check must be selective, not per-call.
Measured overheads from the literature (your latency targets) Deterministic gateways are effectively free: Prompt-CFI 0.04 ms, MCP Guardian 3–4 ms, AEGIS 8.3 ms (regex+schema, 100% block on known attacks), MiniScope 1–6 %, gVisor sandbox 2–5 ms TCP. The small model is the cost center: SentinelAgent's NLI ~20 ms/step; a small generative judge is 50 ms–2 s. ⇒ Run deterministic on 100% of calls, LLM on a small fraction.

Keeping the proxy fast — four levers

  • Tier it. Deterministic contract checks first; escalate to the LLM only on ambiguity or sensitive capability (write / exec / network / PII).
  • Gate vs. audit. Block destructive calls synchronously; for read-only/low-risk calls, let the LLM check run asynchronously (flag after the fact) so it's off the critical path.
  • Cache. Memoize verdicts for repeated (tool, arg-shape, intent) patterns.
  • Right-size the model. A fine-tuned small classifier/NLI (cf. SentinelAgent's 328 MB model) beats a big generative judge on both latency and the Firewall-Fallacy risk.

2. The artifact at the centre: the capability contract

A per-tool, machine-readable description of what the tool is allowed to do, reconciling what it declares (description + schema) against what it actually does (static + dynamic analysis). Sketch:

{
  "tool": "github.create_issue",
  "server": "github-mcp",
  "manifest_hash": "sha256:9f2c…",          // drift / rug-pull anchor
  "declared": {                              // inferred at scan time (LLM + schema)
    "summary": "creates a GitHub issue",
    "expected_capabilities": {
      "network":  ["api.github.com"],        // allowed egress
      "filesystem": "none",
      "process_exec": "none",
      "secrets":  ["GITHUB_TOKEN"]
    }
  },
  "observed": {                              // from static sink analysis + sandbox probe
    "network": ["api.github.com"],
    "filesystem": "none",
    "process_exec": "none"
  },
  "divergence": [],                          // empty ⇒ behaves as declared
  "risk": "low",
  "enforcement": {                           // handed to the proxy
    "sandbox_profile": { "egress_allow": ["api.github.com"], "fs": "deny", "exec": "deny" },
    "arg_schema": { "...": "JSON Schema" },
    "intent_check": "on_sensitive"           // when to spend the LLM
  }
}

Divergence is the core signal: a "calculator" whose observed.network is non-empty, or a "read" tool with observed.filesystem: write, is flagged regardless of how the payload is coded — this is what defeats the obfuscation that beats pattern-matchers (MCP-ITP at 0.3% detection).

3. How the scanner works (Hauptprojekt)

  1. Enumerate. Speak MCP (JSON-RPC 2.0 over stdio / Streamable HTTP), do the handshake, list tools/resources/prompts and pull their manifests + schemas.
  2. Declared profile. Map each tool's (description + input/output schema) → an expected capability profile. Small LLM + schema parsing; offline, so cost is irrelevant.
  3. Observed profile. (a) Static: parse server source, find sinks (network/FS/exec/SQL) and taint from tool params — the VIPER/mcp-sec-audit technique. (b) Optional dynamic: run the tool in an instrumented sandbox (seccomp/eBPF/Docker) with probe inputs, record syscalls.
  4. Reconcile. Declared vs observed ⇒ divergence findings + a pinned contract artifact.
  5. Also flag the static-only threats (typosquatting, PMA phrasing, cross-tool refs, plaintext secrets, vulnerable deps) in the same pass — see the scanner spec.
  6. Emit a report (human) + the contract JSON (machine, for the proxy).

4. How the proxy works (Masterarbeit)

A man-in-the-middle on the MCP channel (MCP Guardian already shows this by overriding the tool-invocation call; gateway/proxy products exist). Per tool call:

client ──tool call──▶ [ PROXY ] ──▶ server (sandboxed per contract)
                         │
   1. deterministic (≈ms, every call):
      • manifest hash == pinned?        → drift / rug-pull
      • args valid vs contract schema?  → out-of-scope params
      • op within capability envelope?  → enforced by sandbox profile
   2. semantic (small LLM, selective):
      • intent-consistency: does this call match the user's request?   (catches MCP-ITP)
      • response screening: injection hidden in tool output?           (catches IPI)
   3. decision: allow / block / human-review  +  append to audit log

The three things the proxy adds that the scanner structurally cannot:

  • Drift / rug-pull detection — compare the live manifest hash to the pinned one; subscribe to MCP's tools/list_changed notification and re-derive on change. (verify the exact notification name in the current spec.)
  • Intent-consistency — the small-LLM check that catches a legitimate high-privilege tool being driven against the user's request (the MCP-ITP class static scanners miss at 0.3%).
  • Output screening — treat tool responses as untrusted and scan for indirect prompt injection.

5. Are we capable of this? — honest feasibility

Yes, because every brick already has a working precedent — you're composing known techniques in a novel configuration (which is exactly the right shape for a thesis), not inventing primitives.

Tractable (low risk)MediumHard (the real risks)
Proxying MCP (JSON-RPC, SDKs exist) · manifest/schema parsing · schema validation · hash-pinning & drift · lexical metadata scans · building a labeled corpus from MCPTox/MSB · measuring P/R + latency LLM "description → expected capability profile" (prompt + eval) · intent-consistency prompt design · seccomp/Docker sandbox for dynamic probing · static sink/taint analysis (lean on CodeQL/Bandit) Dynamic execution of tools needing live credentials / external services (VIPER & mcp-sec-audit both hit this wall) · keeping false-positives low on capability derivation · adversarial robustness
De-risked MVP (so the project is bounded) Don't try to cover all 16 threats or all tools. Scope to: (1) Python MCP servers, (2) three capability classes — network, filesystem, process-exec, (3) the high-value threats — tool poisoning, over-privilege, rug-pull, implicit-poisoning/intent. For the sandbox-credential wall, start with mocked / recorded dependencies and tools that run offline; solving the live-credential case cleanly is itself a contribution, not a prerequisite.

6. The Hauptprojekt / Masterarbeit split

The contract is the interface: HP produces it, MA consumes it. Clean dependency, shared threat model, complementary halves.

Hauptprojekt — the ScannerMasterarbeit — the Proxy
ModeStatic, offline, pre-deploymentRuntime, online, on the call path
LatencyIrrelevantCore engineering constraint
DeliverableCLI that derives a capability contract + flags declared-vs-observed divergence + the static threatsProxy that enforces the contract, detects drift/rug-pull, adds intent-consistency + output screening
CatchesPoisoning text, over-privilege, command-injection sinks, typosquatting, secrets, vuln depsRug-pulls, indirect prompt injection, implicit poisoning (intent), out-of-scope params at runtime
EvaluationDetection P/R/F1 on poisoned-vs-benign corpus. Small results OK — originality is the divergence framing + contract emission.Attack-resistance on MCPTox/MSB + implicit-poisoning subset; utility on MCP-Universe; NRP + latency overhead.
Main riskCapability-derivation accuracy; the sandbox wallLatency of the semantic layer; false positives breaking real workflows
Why this is a strong thesis shape HP is bounded (offline, no enforcement, no latency) → deliverable with modest results, yet original (no one derives a behavioral contract + checks declared-vs-observed). MA inherits a working scanner and tackles the harder, empirical, latency-bound enforcement + the runtime-only threats → enough substance for the thesis. One threat model, one artifact (the contract) connecting them.

7. What we actually build (it's a lot)

"Reuse" earlier meant algorithms/libraries, not software. The scanner and the proxy are original systems written from scratch — they don't exist today. We reuse a base LLM, a sandbox primitive, an MCP SDK, and a taint engine the way a compiler reuses a parser: as bricks. Everything that makes this a behavioral-contract MCP security tool is ours.

Minimum viable contribution (so you're never blocked) The smallest HP that is still a real, novel, demonstrable system: MCP harness + declared-capability extractor + static observed-capability + reconciliation/divergence + contract emitter + a small eval. No dynamic sandbox required. That alone is original software that does not exist — everything else (dynamic probing, the whole proxy) is additive depth for the Masterarbeit.
The contribution, stated for an examiner Not "a new model" but "a system + a method": behavioral capability contracts for MCP (declared-vs-observed divergence) and contract-grounded per-call intent-consistency — built from scratch — plus the first empirical evidence of whether it catches the attacks static scanners miss, at what latency and false-positive cost. The system is the deliverable.
